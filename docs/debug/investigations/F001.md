## Issue

**F001 — Tabbing out and back in disables zoom**

After losing and regaining window focus (Alt+Tab), scroll-wheel zoom stops working. Zoom was
functional before the focus round-trip. Suspected cause: input state or VIF state not
reset/restored correctly on WindowEvent::Focused transitions.

**Repro:** Launch app. Scroll-wheel zoom in/out to confirm it works. Alt+Tab away. Alt+Tab back.
Attempt scroll-wheel zoom — broken.

## Phase 1 — Root Cause Investigation

**E1.1** (`emGUIFramework.rs:981-991`): `WindowEvent::Focused(focused)` handler calls
`win.view_mut().SetFocused(&mut tree, focused)` and `win.set_focus_changed()`, but does NOT
clear pressed keys from `self.input_state` on focus loss.

**E1.2** (`emViewInputFilter.rs:822-824`): `emMouseZoomScrollVIF::filter` accepts wheel events
only when `state.IsNoMod() || state.IsShiftMod()`. If Alt is stuck in pressed set, both return
false and wheel events are silently dropped.

**E1.3** (`emInputState.rs:86-87`): `IsNoMod()` returns `false` if ANY modifier (Shift, Ctrl,
Alt, Meta) is in the pressed set. With Alt stuck, `IsNoMod()` is always false after Alt+Tab.

**E1.4** (`emX11WindowPort.cpp:843-853` / `emX11Screen.cpp:609-620`): C++ X11 port calls
`Screen.UpdateKeymapAndInputState()` on `FocusIn`, which calls `XQueryKeymap` to re-sync
the keyboard state from the actual hardware. This clears stuck keys like Alt that were pressed
when focus was lost.

**E1.5** (`emGUIFramework.rs:1050-1060`): Key press/release events update `self.input_state`
(press → insert, release → remove). The release for Alt is not delivered after the window manager
intercepts Alt+Tab, leaving Alt stuck in `self.input_state.pressed`.

**Root cause chain:**
1. Scroll-wheel zoom stops after Alt+Tab
2. ← VIF rejects wheel events: `state.IsNoMod()` is false (E1.2, E1.3)
3. ← `Alt` is stuck pressed in `self.input_state.pressed`
4. ← `input_state` not cleared on `WindowEvent::Focused(false)` (E1.1)
5. ← OS window manager intercepts Alt+Tab; Alt release not delivered to our window
6. ← C++ fixes this via `XQueryKeymap` on FocusIn (E1.4); Rust has no equivalent

## Phase 2 — Pattern Analysis

**C++ pattern** (`emX11WindowPort.cpp:843-853`):
- `FocusIn` handler calls `Screen.UpdateKeymapAndInputState()` 
- `UpdateKeymapAndInputState()` calls `XQueryKeymap` to read actual physical key state
- If keymap changed, `UpdateInputStateFromKeymap()` re-syncs `InputState` from hardware
- Net effect: stuck modifier keys (Alt etc.) cleared when focus returns

**Rust difference:**
- `WindowEvent::Focused(true/false)` handler only calls `SetFocused` and `set_focus_changed`
- No clearing or re-syncing of `self.input_state` on focus changes
- winit does not automatically deliver release events for keys pressed before focus loss

**Fix approach:** Clear all pressed keys from `self.input_state` on `WindowEvent::Focused(false)`.
This is the standard defensive pattern used by GUI frameworks when the OS doesn't guarantee
key-release delivery. Simpler than querying the actual hardware state (X11-only and unnecessary
since we just need to clear the stuck state).

## Phase 3 — Hypotheses

HYPOTHESIS: Clearing all pressed keys from `self.input_state` on `WindowEvent::Focused(false)` 
will prevent Alt from getting stuck, making `state.IsNoMod()` true again after focus return,
so wheel zoom events are accepted by the VIF.
OUTCOME: CONFIRMED (implemented in Phase 4 and verified via cargo check + clippy + nextest)
EVIDENCE: E1.1, E1.2, E1.3 (root cause); E1.4 (C++ approach); E1.5 (mechanism)

## Root Cause

`emGUIFramework` does not clear `self.input_state` on `WindowEvent::Focused(false)`. When the
user Alt+Tabs out, the Alt key press is recorded in `input_state.pressed` but the Alt release
is swallowed by the window manager and never delivered. On focus return, `input_state` still
shows Alt as pressed. `emMouseZoomScrollVIF::filter` checks `state.IsNoMod() || state.IsShiftMod()`
before processing wheel events — with Alt stuck, both checks fail and all wheel zoom events are
silently dropped.

**Fix:** Add `self.input_state.clear_keys()` to the `Focused(false)` branch of the
`WindowEvent::Focused` handler. Add `clear_keys()` method to `emInputState` that clears the
`pressed` set (but preserves mouse position).
