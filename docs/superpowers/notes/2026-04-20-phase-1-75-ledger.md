### Drift-prevention notes (carry-forward into Task-dispatch prompts)

1. Explicit grep assertion at end of each task (e.g., "confirm `rg -w 'sub_scheduler' crates/` returns zero before marking complete").

3. Pre-commit `--no-verify` usage permitted ONLY for checkpoint commits within a single task; final task commit must pass full gate.

The user's standing rule is in CLAUDE.md: "The entire goal of this plan is to CLEAN UP divergences, workarounds, and hacks; not to create new ones." All task prompts should echo this.

#### Task 2 — Scheduler dispatch rewrite (2026-04-20)

Wired `TreeLocation` through the scheduler: every `register_engine` site now tags engines with a location, and `DoTimeSlice` walks that location per engine before constructing `EngineCtx`.

Files modified (core):
- `crates/emcore/src/emScheduler.rs` — `EngineCtxInner` gains `engine_locations: SecondaryMap<EngineId, TreeLocation>`. `register_engine` signature grows a `tree_location: TreeLocation` param and populates the map. `remove_engine` clears it. A free helper `dispatch_with_resolved_tree(outer_tree, &loc, f)` walks the `TreeLocation` chain via `take_behavior` / `as_sub_view_panel_mut` / `sub_tree_mut` and puts behaviors back on return. `DoTimeSlice` clones the per-engine location and invokes the helper before calling `behavior.Cycle(ctx)`.
- `crates/emcore/src/emEngineCtx.rs` — `ConstructCtx::register_engine` trait method gains `tree_location: TreeLocation`. `EngineCtx::register_engine`, `SchedCtx::register_engine`, and the three `ConstructCtx` impls (for `EngineCtx`, `SchedCtx`, `InitCtx`) all forward the new param.

Files modified (call sites — ~60 sites total): every existing call now passes `TreeLocation::Outer` (Task 2 preserves outer-only registration — Task 3 introduces `SubView` registration at sub-view panel construction time). Production: `emView::RegisterEngines` / popup recv engines, `emPanelTree::create_child` adapter registration, `emPriSchedAgent`, `emMiniIpc`, `emMainWindow`, `emFileManControlPanel` control-panel dummy, `emSubViewPanel::sp4_5_fix_1_timing_sub_scheduler_baseline_slices`. Tests: all `emScheduler::tests::*`, `tests/unit/scheduler.rs`, `tests/integration/{lifecycle,signals}.rs`, `tests/golden/scheduler.rs`.

Nested-dispatch test: added `emScheduler::tests::task2_dispatch_walks_depth_2_subview_location`. Depth-2 fixture — outer tree installs an `emSubViewPanel` behavior at a child panel; that panel's `sub_tree` installs a second `emSubViewPanel` behavior. A probe engine registered with `TreeLocation::SubView(outer_child, SubView(inner_child, Outer))` records `ctx.tree as *mut PanelTree` during `Cycle`. The test asserts the recorded pointer equals a pre-dispatch snapshot of the innermost `sub_tree_mut` pointer, proving the two-hop take/put walk reaches the correct tree. Passes.

Test-only concession: the depth-2 test leaks the nested `emSubViewPanel` behaviors via `std::mem::forget` at teardown. Reason: `emSubViewPanel::new` constructs its own `sub_scheduler` with registered `UpdateEngineClass`/`VisitingVAEngineClass` engines; properly tearing them down mid-test requires the full `emSubViewPanel` teardown helper (which lives inside `emSubViewPanel.rs` tests and depends on `sub_view.update_engine_id`). Task 3/4 will delete `sub_scheduler` entirely, at which point this teardown concession goes away. This is test-plumbing only — below the observable surface; not a divergence.

Borrow-checker: the dispatch walk is a small recursive function over `&TreeLocation`, not an explicit stack. The `take_behavior`/`put_behavior` pairing is naturally lexically scoped by each recursive call — no issue with nested borrows. Cloning the `TreeLocation` (shallow `PanelId` + `Box<TreeLocation>` chain) out of `engine_locations` before the walk avoids aliasing the scheduler's inner state during dispatch; clone cost is one `Box::new` per `SubView` hop, which matches the rare-path cost profile.

Naming/file: no renames, no new files. `as_sub_view_panel_mut` (added in Task 1) is the only downcast surface; no `Any` / `dyn Any` involved in the dispatch path.

Gate: `cargo fmt` applied, `cargo clippy --all-targets --all-features -- -D warnings` clean, `cargo nextest run` = 2456 passed / 0 failed / 9 skipped (one more than the 2455 baseline — the new `task2_dispatch_walks_depth_2_subview_location` test).
