# Adaptive Interpolation AVX2 Matching Rewrite

Date: 2026-04-14

## Summary

Rewrite Rust adaptive interpolation to match C++ AVX2 `InterpolateImageAvx2Adaptive`
output exactly. Currently matches the C++ scalar `InterpolateImageAdaptive`, which
produces different results due to different factor scaling, premultiplication timing,
and per-term rounding.

## Problem

7 golden tests fail at zero tolerance. All involve adaptive interpolation:
- `tktest_1x` (1104px, max_diff=53) — border images via `sample_adaptive_premul_fp_section`
- `tktest_2x` (1px, max_diff=2) — same
- `testpanel_root` (543px, max_diff=76) — border images + colored image
- `testpanel_expanded` (8070px, max_diff=234) — borders at scale
- `painter_image_scaled` (493px, max_diff=1) — direct image interpolation
- `listbox` (223px, max_diff=1) — font glyph via `sample_adaptive_lum_section`
- `textfield` (58px, max_diff=1) — font glyph via `sample_adaptive_lum_section`

Golden data is generated by the C++ AVX2 path. Rust currently implements the scalar path.

## Root Cause: Five Arithmetic Differences

### 1. Factor Table Scale

Scalar: 1024-scaled positive factors. `fv1 = round((2o³-3o²+1) * 1024)`.
AVX2: -32768-scaled negative factors. `fv1 = round((2o³-3o²+1) * (-32768))`.

### 2. Premultiplication Timing

Scalar: `premul = pixel * alpha` (product range 0-65025), divide by 255 later via FINPREMUL.
AVX2: `premul = fast_div255(pixel * alpha) << 5` (range 0-8160), no later division needed.

### 3. Interpolation Arithmetic

Scalar: `result = v1*fv1 + v2*fv2 + s1*fs1 + s2*fs2` — single multiply-accumulate in i32/i64.
AVX2: `result = mulhrs(v1,fv1) + mulhrs(v2,fv2) + mulhrs(s1,fs1) + mulhrs(s2,fs2)` where
`mulhrs(a,b) = (a*b + 0x4000) >> 15`. Per-term rounding introduces ±1 differences.

### 4. Inter-pass Processing

Scalar: `FINPREMUL_SIGNED_COLOR` between Y and X passes: `rgb = (rgb + 0x7f) / 0xff`.
AVX2: No inter-pass processing (premul division already done at pixel load time).

### 5. Output Scaling

Scalar: `(value + 0x7FFFF) >> 20`.
AVX2: `(value + 0x10) >> 5`, then `packus_epi16` saturation, then `min(rgb, alpha)`.

## Design: Double-Negation Pipeline

The AVX2 factor table is all-negative (because +32768 overflows i16). This means one
interpolation pass negates values. With two passes (Y then X), the double negation
cancels out:

1. Y-pass: positive input × negative factors → negative intermediate
2. X-pass: negative input × negative factors → positive output

The sign-negation trick inside `InterpolateFourVectorsAdaptive` handles direction:
- Compute `neg = if v2 > v1 { -1 } else { 1 }`
- Apply sign to all values (normalizes to v2 ≤ v1)
- Slopes, peaks, interpolation all work on normalized direction
- Restore sign at end

This works correctly even when inputs are negative (from Y-pass), because the cmpgt
comparison on negative values reverses the ordering naturally.

## Changes

### File: `crates/emcore/src/emPainterInterpolation.rs`

#### 1. Replace `adaptive_factors()` with `avx2_adaptive_factors()`

Delete the 1024-scaled `[[i32; 4]; 257]` table. Replace with the -32768-scaled
`[[i16; 4]; 257]` table copied from C++ `AdaptiveFactorsTable`.

#### 2. Add `mulhrs(a: i32, b: i32) -> i32`

```rust
fn mulhrs(a: i32, b: i32) -> i32 {
    (a * b + 0x4000) >> 15
}
```

#### 3. Rewrite `interpolate_four_values_adaptive`

Change signature from `(v0: i32, v1: i32, v2: i32, v3: i32, o: u32) -> i64`
to `(v0: i32, v1: i32, v2: i32, v3: i32, o: u32) -> i32`.

Replace branchy scalar slope/peak logic with AVX2-matching branchless version:

```
sign = if v2 > v1 { -1 } else { 1 }
apply sign to v0..v3
compute slopes s01, s12, s21, s23
compute adaptive slopes s1, s2 using min/max (branchless)
clamp s1, s2 to non-positive
peak suppression on v1, v2
result = mulhrs(v1,fv1) + mulhrs(v2,fv2) + mulhrs(s1,fs1) + mulhrs(s2,fs2)
restore sign
```

All arithmetic in i32 to match mulhrs precision without i16 overflow concerns.

#### 4. Rewrite `sample_adaptive_premul_fp_section`

Current pipeline:
```
pixel * alpha → interp_y(1024 factors) → FINPREMUL(/255) → interp_x(1024 factors) → >>20
```

New pipeline:
```
fast_div255(pixel * alpha) << 5 → interp_y(mulhrs, -32768 factors) → interp_x(mulhrs, -32768 factors) → +0x10 → >>5 → clamp
```

Store Y-interpolation results as i16 (truncate to 16-bit) to match C++ intermediate storage.

#### 5. Rewrite `sample_adaptive_lum_section`

For 1-channel (no premul needed):
```
pixel << 5 → interp_y(mulhrs, -32768 factors) → interp_x(mulhrs, -32768 factors) → +0x10 → >>5 → clamp
```

Same interpolation function, just different input preparation.
