## W5a — Phase-8 close-signal test: single-engine promotion

### Current state

`test_phase8_popup_close_signal_zooms_out` at `crates/emcore/src/emView.rs:6105` exercises the popup close-signal drain, but does so across **two engines**:

1. A real `WakeUpUpdateEngineClass` proves the wake-up connection (`get_signal_refs(close_sig, eng_id) >= 1`).
2. The test then swaps in a `NoopEngine`, disconnects the close_signal from both engines, manually calls `sched.borrow_mut().fire(close_sig)`, and drives `DoTimeSlice` so `sig.clock` advances past the dummy engine's clock. `v.Update(&mut tree)` is then called directly.

The reviewer's finding (carried in the closeout doc): assertions hop between two engines in a way that reads as a harness hack, not an integrated run.

### Target state

One real engine, one integrated drive loop. No `NoopEngine`, no manual `sched.fire(close_sig)`, no disconnect/reconnect gymnastics. The test proves the full real chain:

```
popup close API → close_signal fires → WakeUpUpdateEngine wakes →
  scheduler DoTimeSlice → emView::Update drains NoticeList and sees
  close_signal signaled → ZoomOut → popup teardown
```

### Design

**Location:** test stays inline in `crates/emcore/src/emView.rs` under `#[cfg(test)]`. The module already has `pub(crate)` access to `PanelTree::get_mut` and the rest of the internal surface, so no visibility widening is needed.

**Rejected alternatives:**
- Widening `PanelTree::get_mut` to `pub` — violates `CLAUDE.md` visibility rule (`pub(crate)` default; `pub` only for public API). The reviewer finding is a test-shape concern, not a consumer-surface concern.
- Adding a `#[cfg(test)]` test-support method on `PanelTree` — unnecessary when the existing access path works.

**Test flow:**

1. Build tree, view, scheduler; `attach_to_scheduler` (as today).
2. Clear `zoomed_out_before_sg`, set `POPUP_ZOOM`, `RawVisit` to create the popup (as today).
3. Verify the wake-up connection the same way as today: `sched.borrow().get_signal_refs(close_sig, eng_id) >= 1`. This assertion stays; it is cheap and orthogonal to the drive-loop change.
4. **Replace the engine-swap block** with a real close trigger + bounded drive loop:
   - Trigger close via the public popup API. Identify the intended method during implementation — the first candidate is `PopupWindow.borrow().Signal(close_signal)` / `signal_close()` / whatever `emView` exposes for a popup consumer to ask the view to close its popup. If no such public path exists, fall back to `sched.borrow_mut().fire(close_sig)` and **document inline** that this is the minimum-surface substitute for a missing consumer API (this is itself a port gap to be noted, not hidden).
   - Drive the scheduler with a bounded loop (`for _ in 0..N { sched.borrow_mut().DoTimeSlice(&mut tree, &mut windows); if v.PopupWindow.is_none() { break; } }`). `N = 8` is the starting budget; adjust during implementation if empirically one `DoTimeSlice` completes the chain.
5. Assert `v.PopupWindow.is_none()` after the loop. Assert the loop terminated early (not on exhaustion) so exhaustion failures are distinguishable from tear-down successes.
6. Same scheduler cleanup as today: remove engine, remove signals, clear `eoi_engine_id`, `visiting_va_engine_id`.

**Phase structure:** one plan phase. Expected footprint: ~30 lines changed in one test function; no production code changes.

**Acceptance:**
- `cargo-nextest ntr` passes with no `NoopEngine` reference in `emView.rs`.
- Test asserts across exactly one engine registration (the real `WakeUpUpdateEngineClass`).
- `grep -n "NoopEngine" crates/emcore/src/emView.rs` returns zero matches.
- No new `#[cfg(test)]` helper methods on `PanelTree` or `emView`.
- Pre-commit hook (fmt + clippy + nextest) passes.

### Non-goals

- Moving the test out of `emView.rs`.
- Changing any non-test production code.
- Restructuring the close-signal API surface beyond documenting a call-site gap if one is found.

## W5b — NoticeList dispatch divergence: classify and defer

### Current state

`crates/emcore/src/emGUIFramework.rs:509-514` computes a single `pixel_tallness` for the whole frame by picking an arbitrary window:

```rust
let pixel_tallness = self
    .windows
    .values()
    .next()
    .map(|rc| rc.borrow().view().GetCurrentPixelTallness())
    .unwrap_or(1.0);

self.tree.run_panel_cycles(pixel_tallness);
let had_notices = self.tree.HandleNotice(window_focused, pixel_tallness);
```

This is architecturally wrong once more than one window is active: each view has its own `CurrentPixelTallness` (see `emView.h:216`, set in `emView.cpp:1266`), and C++ dispatches `HandleNotice` **per-view** from `emView::Update()` (`emView.cpp:1312`). The Rust port currently dispatches once globally against a shared `PanelTree`, so there is no principled way to select a single pixel_tallness.

### Root-cause divergence (for the record)

`emPanelTree.rs:291-296` already carries a `DIVERGED:` note acknowledging that Rust has one global NoticeList (Vec-backed) where C++ has a per-view `PanelRingNode*` intrusive list. The existing note classifies the **data-structure** divergence (`PanelRingNode*` sentinel → `Vec<PanelId>`) but does not classify the **dispatch-driver** divergence (C++ drives per-view from `emView::Update`; Rust drives globally from `emGUIFramework`).

Per the Port Ideology:
- **Storage shape (global `PanelTree`):** Forced. Rust ownership rules make per-view intrusive ring nodes impractical without re-introducing `unsafe` and custom allocators.
- **Data structure (`Vec` vs intrusive ring):** Idiom adaptation. Below observable surface.
- **Dispatch driver (global framework call vs per-view `emView::Update`):** **Design-intent violation.** C++ dispatch timing is observable (order of notice handling across views, pixel_tallness used per view). This is the load-bearing part of the divergence.

The correct fix is architectural: move `NoticeList` ownership from `PanelTree` to `emView`, dispatch from `emView::Update`, use each view's own `CurrentPixelTallness`. That is not small and is not in W5b's scope.

### Target state for W5b

W5b is **documentation-only**. No behavioral change, no golden diff, no new tests.

### Design

Three edits:

1. **Expand `DIVERGED:` at `crates/emcore/src/emPanelTree.rs:291-296`.** Keep the current data-structure explanation. Add a second paragraph classifying the divergence axes per the Port Ideology, explicitly naming the third axis (dispatch driver) as the design-intent violation. Reference `emView.cpp:1312` as the correct dispatch site. Reference the successor workstream by name.

2. **Add a TODO at `crates/emcore/src/emGUIFramework.rs:509-514`**, one block comment immediately above the `let pixel_tallness = …` line. Exact content:

   > `TODO(per-view-notice-dispatch): selecting one window's pixel_tallness for the whole frame is wrong when multi-window is active. C++ dispatches HandleNotice per-view from emView::Update (emView.cpp:1312), using that view's own CurrentPixelTallness. Correct fix requires moving NoticeList ownership from PanelTree to emView. Tracked as: "Per-view notice dispatch (emView.cpp:1312 parity)" in the emView subsystem closeout doc.`

3. **Register the successor workstream in `docs/superpowers/notes/2026-04-18-emview-subsystem-closeout.md` §8** (residual debt). Add an entry named **"Per-view notice dispatch (emView.cpp:1312 parity)"** with this scope sketch:
   - Move `NoticeList` ownership from `PanelTree` to `emView`.
   - Dispatch `HandleNotice` from `emView::Update`, once per view per frame, using the view's own `CurrentPixelTallness`.
   - Requires panel→view ownership in `PanelTree` (or equivalent: walking each view's subtree from its root panel to partition notices).
   - `run_panel_cycles` is a separate Rust-only construct (C++ panels self-register as engines via `emEngine` inheritance, which the Rust port does not mirror) and stays out of this workstream.

**Phase structure:** one plan phase, three edits. Expected footprint: ~15 lines of comments/doc changes across three files.

**Acceptance:**
- `grep` for `TODO(per-view-notice-dispatch)` returns exactly one match (the new comment).
- `emPanelTree.rs:291-296` contains the expanded divergence classification.
- The closeout doc §8 lists the successor workstream by the name above.
- `cargo check` and `cargo clippy -- -D warnings` clean (no functional change should trigger any).
- Pre-commit hook passes.

### Non-goals (explicitly out of scope for W5b)

- Moving `NoticeList` to `emView`.
- Adding panel→view ownership in `PanelTree`.
- Changing any dispatch timing.
- Touching `run_panel_cycles`.
- Any change to `emGUIFramework::about_to_wait` beyond the one added comment.

These all belong to the successor workstream registered in step 3.
