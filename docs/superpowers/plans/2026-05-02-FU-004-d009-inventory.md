# FU-004 D-009 Polling-Intermediary Inventory Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce a verified, classified row-table inventory of every D-009 polling-intermediary candidate in the Rust tree, append it to (or alongside) the FU-004 bucket file, and add minimum in-place doc-comment / annotation / scratch-dump entries demanded by the classification.

**Architecture:** Research-and-classification work — not feature code. The pipeline is **enumerate → verify per-candidate against C++ → publish row table + in-place adjustments**. TDD's red-green-refactor loop does not apply directly; instead, each phase produces a durable artifact (raw candidate list, per-candidate verification notes, final row table). Each task is small enough to commit independently. Acceptance is mechanical: every enumerated candidate has a row, every classification cites file:line evidence, every action listed in the spec's deliverable table is reflected either in source (comment/annotation) or in scratch (dump entry) or in a follow-up spec stub (D-007 candidates).

**Tech Stack:** ripgrep (`rg`) for enumeration, manual reads of Rust + C++ source for verification, Markdown for the deliverable. No code change to compiled crates. `cargo xtask annotations` and `cargo-nextest ntr` are the gate at the end (annotations gate is load-bearing if any `DIVERGED:` block is added).

**Non-interactive defaults chosen for this plan (would normally be asked):**
1. **Inventory location.** Append to the existing bucket file `docs/debug/audits/2026-04-27-signal-drift-tier-b/remediation/followups/FU-004-d009-polling-sweep.md` if the row table fits in roughly 80 lines, otherwise spill to a sibling `FU-004-inventory.md` and link from the bucket file. The threshold is judged when the row table is drafted in Task 9.
2. **Scratch entries for vestigial candidates** go into the existing `docs/scratch/2026-05-02-future-work-dump.md` (already present), under a new `## FU-004 vestigial sweeps` section.
3. **`D-007 candidate` stubs** are recorded as a single subsection at the bottom of the inventory and as a one-line entry in the scratch-dump pointing to "follow-up spec required". A separate spec is **not** written by this plan.
4. **Verification depth budget.** Per the spec's risk note, any candidate whose verification grows past one focused reading session is classified `needs deeper audit` and split into its own future bucket reference; this plan does not chase emTextField-cluster setter-mapping into a second day of work.

## Task 8: Per-candidate verification (loop)

**Files:**
- Modify: `target/fu-004-scratch/candidates.md` (fill in row data).

This task is a fixed-process loop applied independently to each candidate from Task 7. Every candidate gets exactly the per-candidate process specified in the spec's Unit 2. The number of iterations equals the number of surviving rows.

- [ ] **Step 1: For each candidate, read declaration context**

For row `<file>:<line>`, run:
```
rg -B10 -A5 -n . <file> | sed -n '<line-10>,<line+5>p'
```
or simply `Read` the file at `offset = line - 12, limit = 25`. Record any existing doc-comment (cited C++ file:line, audit ID, or rationale) into the `Evidence` column.

- [ ] **Step 2: Enumerate setters**

Run:
```
rg -n 'self\.<field>\s*=|self\.<field>\.set|self\.<field>\.replace\(true|self\.<field>\.replace\(Some' \
  /home/alex/Projects/eaglemode-rs/crates/<crate>/src/
```
List file:line setters in `Setters` column.

- [ ] **Step 3: Identify drain site**

From Task 6's `cycle-bodies.txt` cross-reference, plus a fallback grep:
```
rg -n '<field>\.take\(\)|<field>\.replace\(false\)|<field>\.replace\(None\)|<field>\.set\(false\)|<field>\.set\(None\)' \
  /home/alex/Projects/eaglemode-rs/crates/<crate>/src/
```
Record file:line in `Drain` column.

- [ ] **Step 4: Read C++ counterpart**

From the doc-comment cite, or if absent:
```
rg -n '<FieldName>' ~/Projects/eaglemode-0.96.4/include/emCore/ ~/Projects/eaglemode-0.96.4/src/emCore/
```
Open the cited C++ file and read the surrounding ~30 lines. Record file:line in `C++ counterpart`. If no analogue exists, write `none`.

- [ ] **Step 5: Determine C++ shape**

Read enough C++ to decide one of:
- `same polling shape` — C++ has equivalent set-and-cycle-drain.
- `synchronous fire` — C++ fires the side-effect inline at the mutation site (Rust diverged → D-007 candidate).
- `no analogue` — Rust-only.
Record in `C++ shape`.

- [ ] **Step 6: Classify**

Apply the decision table:
- C++ same polling → `C++-mirrored`.
- C++ synchronous → `D-007 candidate`.
- No setter outside test code → `vestigial`.
- Documented in-progress D-007 conversion (e.g., emTextField B3.4c/d) → `in-progress-migration`.
- Owned by another open spec (e.g., emDialog Phase 3.6 Task 4) → `out-of-scope`.
- Forced retention with cited forced category → `forced retention`.

Record in `Classification` plus a one-line `Evidence` citing file:line for the deciding observation.

- [ ] **Step 7: Determine action**

Per the spec's deliverable table:
- `D-007 candidate` → action: "follow-up spec required (recorded in scratch-dump)".
- `C++-mirrored` → action: "add explanatory doc-comment if missing".
- `forced retention` → action: "add `DIVERGED:` annotation with category cite".
- `vestigial` → action: "scratch-dump entry with cleanup trigger".
- `in-progress-migration` → action: "cross-reference owning track".
- `out-of-scope` → action: "cross-reference owning effort".

Record in `Action`.

- [ ] **Step 8: Time-box check**

If verification of this candidate has consumed more than one focused reading session (rough heuristic: 30+ minutes of source reading without a classification), classify as `needs deeper audit` with action "split into own follow-up bucket". Move on.

- [ ] **Step 9: Repeat for next candidate**

Until all rows are filled. Then proceed to Task 9.

- [ ] **Step 10: No commit yet**

Scratch only. The verified table is committed in Task 9.
