# Golden Parity Status

Complete accounting of all divergences between Rust zuicchini and C++ emCore,
with source references, operational guides, and enough context for an LLM
to research and fix any item without prior conversation history.

---

## 1. What is this?

**zuicchini** is a Rust reimplementation of C++ **emCore**, the rendering /
UI-layout kernel of [Eagle Mode](https://eaglemode.sourceforge.net/). The
golden-parity test harness compares zuicchini's pixel output against binary
snapshots ("golden files") captured from the real C++ library.

The harness has **29 painter tests** (pixel rendering) and **13 layout tests**
(rectangle positioning). All 53 tests + 11 scheduler tests currently pass.
0 ignored. 0 failed.

This document catalogues every known divergence: what it is, why it exists,
exactly where in the code it lives, how to reproduce and measure it, and what
it would take to eliminate it.

## 2. How to run things

All commands run from the **workspace root** `/home/ar/Development/egopol`.

```bash
# Run all parity tests (53 tests)
cargo test --test golden_parity -- --nocapture

# Run a single parity test
cargo test --test golden_parity painter_ellipse_basic -- --nocapture

# Run zuicchini unit tests (431 tests)
cargo test -p zuicchini

# Clippy (must be clean)
cargo clippy --workspace -- -D warnings

# Regenerate golden data from C++ (only if gen_golden.cpp changes)
make -C zuicchini/golden_gen run
```

### How to probe a test's actual divergence

Set the test's tolerance to `(1, 0.1)` in `painter.rs`, then run it. It will
fail and print:

```
Image mismatch: 662/65536 pixels (1.01%) exceed tolerance 1, max_diff=250
  (127,28): actual=rgb(0,255,0) expected=rgb(0,190,0)
  ...
```

- **max_diff**: worst single-channel difference across all pixels
- **fail_count/total (fail_pct)**: how many pixels have any channel diff > ch_tol
- **First 10 failures**: pixel coordinates + actual vs expected RGB

Revert the tolerance after probing.

### How `compare_images` works

File: `tests/golden_parity/common.rs:55`

```
compare_images(actual, expected, width, height, channel_tolerance, max_failure_pct)
```

For each pixel, compute `max(|aR-eR|, |aG-eG|, |aB-eB|)`. If that exceeds
`channel_tolerance`, the pixel is a "failure". If the percentage of failing
pixels exceeds `max_failure_pct`, the test fails. Alpha channel is **excluded**
(C++ uses channel 3 for "remaining canvas visibility", not standard alpha).

### Golden file binary format

**Painter:** `golden/painter/{name}.painter.golden`
- Bytes 0-3: width (u32 LE), bytes 4-7: height (u32 LE), bytes 8+: RGBA data

**Layout:** `golden/layout/{name}.layout.golden`
- Bytes 0-3: child count (u32 LE), then child_count x 32 bytes (x,y,w,h as f64 LE)

Generated by `zuicchini/golden_gen/gen_golden.cpp`. **Never modify golden files
or the generator.**

## 3. Glossary

| Term | Meaning |
|------|---------|
| **ch_tol** | `channel_tolerance` — max per-channel (R/G/B) absolute diff allowed per pixel before counting it as a failure |
| **fail_pct** | Percentage of pixels that exceed ch_tol |
| **max_diff** | Largest per-channel diff observed across all pixels (at ch_tol=1) |
| **Raw measurement** | max_diff and fail_pct measured at ch_tol=1 (the strictest useful probe) |
| **Category A** | Algorithmic bug — wrong output, must be fixed in source |
| **Category B** | Tunable rounding — same algorithm, different integer rounding |
| **Category C** | Architectural gap — fundamentally different algorithm |
| **Fixed12** | 12-bit fractional fixed-point type used for sub-pixel coordinates (1 unit = 1/4096 pixel) |
| **Scanline rasterizer** | Converts polygon vertices to per-row horizontal spans with AA coverage |
| **Stroke expansion** | Converts a line/outline + thickness into a filled polygon |

## 6. Category A: Bugs found and fixed

Real implementation bugs that produced wrong output. All fixed. Included here
for historical reference and so an LLM doesn't re-investigate them.

### A1. Radial gradient painted outside ellipse bounds

- **Test:** `gradient_radial`
- **Was:** 25% pixel mismatch. Gradient filled bounding box, not ellipse.
- **Root cause:** Pixel loop iterated full rectangle; pixels outside ellipse
  got `t=1.0` (outer color) instead of being skipped.
- **Fix:** Rewrote `paint_radial_gradient()` (:493) to use
  `ellipse_polygon()` + scanline + `PixelTexture::RadialGradient`, so the
  ellipse boundary is enforced by the polygon scan. Removed dead
  `sample_radial_gradient()` from `interpolation.rs`.
- **Files changed:** `src/render/painter.rs`, `src/render/interpolation.rs`

### A2. Ellipse sector angle convention mismatch

- **Test:** `ellipse_sector`
- **Was:** 36% pixel mismatch. Completely wrong area filled.
- **Root cause:** C++ uses (start_degrees, sweep_degrees). Rust used
  (start_radians, end_radians). Two mismatches stacked.
- **Fix:** Changed `paint_ellipse_sector()` (:387) and
  `paint_ellipse_sector_outlined()` (:1755) to take degrees and sweep.
  Updated callers in `examples/test_panel.rs`, `examples/bench_interaction.rs`,
  `examples/profile_testpanel.rs`.
- **C++ reference:** `emPainter.cpp:1078` — `startAngle *= M_PI/180.0;
  rangeAngle *= M_PI/180.0;`

### A3. Raster layout strict mode didn't center grid

- **Test:** `layout_raster_strict`
- **Was:** C++ centered 167x167 cells; Rust placed 333x333 cells at origin.
- **Root cause:** Alignment was applied to unclamped grid, not post-clamp.
- **Fix:** Restructured `do_layout_inner()` in `src/layout/raster.rs` to
  compute grid aspect ratio from clamped dimensions and apply alignment to
  the surplus axis.
- **C++ reference:** `emRasterLayout.cpp:348-381`

## 7. Category B: Tunable rounding differences

Same algorithm, different integer rounding. 2/255 channel diff.

### B1. Linear gradient interpolation rounding

- **Tests:** `gradient_h` (2, 1.0), `gradient_v` (2, 1.0)
- **Where:** `sample_linear_gradient()` at `interpolation.rs:335` computes
  `color_a.lerp(color_b, t)`. The float→u8 conversion path differs from C++.
- **To fix:** Find the C++ gradient interpolation in `emPainter.cpp` (look for
  `LinearGradientTexture` or the gradient texture sampling loop). Match the
  exact integer multiply-shift-round sequence.
- **Effort:** Small. Likely a one-line rounding change in `Color::lerp()`.

### B2. Canvas-color alpha blend rounding

- **Test:** `canvas_color` (2, 0.5)
- **Where:** `blend_pixel()` at `painter.rs:3418`. The canvas-color path
  (`:3439-3460`) computes final RGB analytically when the background is known.
  The integer arithmetic differs from C++ by 1-2/255.
- **To fix:** Find the C++ canvas-color blend in `emPainter.cpp` (the
  "canvasColor is opaque" fast path). Match the multiply-then-shift order.
- **Effort:** Small. Likely a single integer arithmetic change.

## 8. Category C: Architectural gaps

Fundamentally different algorithms. These are not bugs — the Rust code is
correct, just different. Closing each gap means adding the same specialized
code path that C++ has.

### C1. ellipse-polygon — Polygon approximation of ellipses

| | |
|-|-|
| **C++ does** | Solves ellipse equation per scanline row → exact curve boundary |
| **Rust does** | `ellipse_polygon()` (:3359) → N-segment polygon → `rasterize()` in `scanline.rs:94` |
| **Segment count** | `adaptive_circle_segments()` (:3583): `CIRCLE_QUALITY(4.5) * sqrt(rx*sx + ry*sy)`, clamped 3-256 |

| Test | Tolerance | Raw max_diff | % differ at ch_tol=1 |
|------|-----------|-------------|---------------------|
| `ellipse_basic` | (80, 0.5) | 250 | 1.01% |
| `gradient_radial` | (80, 1.0) | 248 | 26.10% |
| `ellipse_sector` | (80, 0.5) | 225 | 0.29% |

The `gradient_radial` 26% is misleading — the gradient color varies smoothly
across the boundary, so many pixels differ by small amounts. At ch_tol=80,
only ~1% exceed the threshold.

**To close — option A (incremental):** Increase segment count. Double
`CIRCLE_QUALITY` from 4.5 to 9.0. Diminishing returns past ~128 segments
but easy to try. Probe and measure improvement.

**To close — option B (full fix):** Implement a native ellipse scanline path
in `scanline.rs` that computes x-entry and x-exit per row using
`x = cx +/- rx * sqrt(1 - ((y-cy)/ry)^2)`. This eliminates the polygon
approximation entirely for ellipses. Look at `emPainter.cpp` around the
`PaintEllipse` function for the C++ approach.

**Where to start reading C++:** Search `emPainter.cpp` for `ScanlineTool`
and ellipse-specific row computation. The key is how C++ computes the
fractional x-coverage at ellipse entry/exit per scanline row.

### C2. scanline-aa — Anti-aliasing coverage computation

| | |
|-|-|
| **C++ does** | Fixed-point sub-pixel coverage with specific bit width and rounding |
| **Rust does** | `Fixed12` (12-bit frac, `fixed.rs:8`), coverage = `frac * 255 / 4096` in `make_span()` (`scanline.rs:219`) |

| Test | Tolerance | Raw max_diff | % differ at ch_tol=1 |
|------|-----------|-------------|---------------------|
| `polygon_tri` | (73, 0.5) | 73 | 0.93% |
| `polygon_star` | (80, 0.5) | 251 | 1.44% |
| `polygon_complex` | (80, 0.5) | 240 | 1.22% |
| `clip_basic` | (64, 0.5) | 64 | 0.23% |

Differences are confined to a 1-2 pixel band at polygon edges. Visually
imperceptible.

**To close:** Reverse-engineer C++ fixed-point bit width from `emPainter.cpp`
(search for the scanline polygon fill code — look for "alpha" or "coverage"
computation near the edge-crossing logic). Then match the same bit width and
rounding in `Fixed12` and `make_span()`.

**Where to start reading C++:** `emPainter.cpp`, search for the polygon fill
implementation. The key variable is whatever stores sub-pixel coverage at
left/right polygon edges. Check if C++ uses 8-bit, 12-bit, or 16-bit
fractions.

### C3. stroke-expansion — Stroke polygon construction

| | |
|-|-|
| **C++ does** | Builds stroke outline polygons tightly coupled to its scanline rasterizer, with specific offset-curve, corner, and miter geometry |
| **Rust does** | `paint_solid_polyline()` (:2309) builds outline polygon independently, then feeds to `rasterize()` |

| Test | Tolerance | Raw max_diff | % differ at ch_tol=1 |
|------|-----------|-------------|---------------------|
| `line_basic` | (80, 1.0) | 152 | 1.21% |
| `line_dashed` | (80, 2.0) | 255 | 1.90% |
| `outline_rect` | (80, 4.5) | 255 | 4.25% |
| `outline_ellipse` | (80, 2.5) | 255 | 3.02% |
| `outline_polygon` | (80, 1.5) | 255 | 2.46% |
| `outline_round_rect` | (80, 4.5) | 255 | 4.21% |
| `bezier_stroked` | (80, 3.0) | 255 | 3.48% |
| `polyline` | (80, 3.5) | 255 | 4.24% |

This is the largest gap (8 tests). The max_diff=255 means some edge pixels
are fully present in one output and fully absent in the other — the stroke
polygon boundaries are in slightly different positions.

**To close:** This is the hardest gap. The Rust stroke expansion in
`paint_solid_polyline()` (:2309-2423) needs to match C++ exactly:
1. **Offset direction** — which side of the line the stroke polygon starts
2. **Corner geometry** — miter vs bevel cutoff, join shape
3. **Cap geometry** — round cap segment count and positioning
4. **Dash accumulation** — how dash/gap lengths are measured along the path

**Where to start reading C++:** Search `emPainter.cpp` for "Stroke" or
"outline" functions. The stroke expansion is likely near `PaintLine`,
`PaintPolygonOutline`, or a shared helper. Focus on how C++ computes the
normal offset at each vertex and how it handles joins.

**Incremental approach:** Start with `outline_rect` (simplest geometry —
4 corners, axis-aligned). Compare the Rust outline polygon vertices against
what C++ would produce for a 200x150 rect with stroke width 3. Fix the
offset direction and corner geometry for that case, then generalize.

### C4. curve-polygon — Bezier curve flattening

| | |
|-|-|
| **C++ does** | Evaluates bezier equation directly during scanline fill |
| **Rust does** | `tessellate_cubic()` (:3643) recursive subdivision → polygon → `rasterize()` |

| Test | Tolerance | Raw max_diff | % differ at ch_tol=1 |
|------|-----------|-------------|---------------------|
| `bezier_filled` | (80, 4.5) | 255 | 4.41% |

**To close — option A (incremental):** Tighten `BEZIER_FLATNESS` from 0.5 to
0.1 or lower. This produces more polygon segments, closer to the true curve.
Measure improvement. Check if `BEZIER_MAX_DEPTH=10` becomes a bottleneck.

**To close — option B (full fix):** Implement direct bezier scanline
integration — compute bezier x-intersections per row analytically (cubic
root solving). Complex but eliminates the flattening entirely.

**Where to start reading C++:** Search `emPainter.cpp` for bezier or cubic
curve handling in the scanline fill path.

### C5. interpolation — Image scaling filter

| | |
|-|-|
| **C++ does** | Likely area-averaged or custom filter for upscaling |
| **Rust does** | `paint_image_full()` (:654) uses nearest for 1:1; `paint_image_scaled()` (:1104) uses `sample()` from `interpolation.rs:316` which auto-selects bilinear/area |

| Test | Tolerance | Raw max_diff | % differ at ch_tol=1 |
|------|-----------|-------------|---------------------|
| `image_scaled` | (70, 0.5) | 118 | 30.68% |

30% of pixels differ but almost all by tiny amounts (max_diff=118, and at
ch_tol=70 only 0.18% remain). This is a filter choice difference, not a bug.

**To close:** Read the C++ image painting code in `emPainter.cpp` (search for
`PaintImage` or `PaintBorderImage`). Identify the interpolation kernel. Then
either add it as an option in `interpolation.rs` or adjust the existing
`sample_bilinear()` (:56) to match.

**Key detail:** The test uses `paint_image_full()` which scales a 64x64 image
to 200x200 (3.125x upscale). Check if C++ uses bilinear with different
coordinate mapping (pixel-center vs pixel-edge) rather than a different filter.

### C6. stroke-ends — Stroke end decoration geometry

| | |
|-|-|
| **C++ does** | Constructs 17 end decoration types as part of stroke pipeline |
| **Rust does** | `paint_stroke_end()` (:2760-3128) with independently implemented geometry per type |

| Test | Tolerance | Raw max_diff | % differ at ch_tol=1 |
|------|-----------|-------------|---------------------|
| `line_ends_all` | (80, 17.0) | 255 | 19.91% |

**ABOVE 5% BUDGET.** This is the worst test. 17 end types in one 256x256
image, each with slightly different geometry, sizing, and positioning vs C++.

The 17 types defined in `stroke.rs:21`:
`Butt, Cap, Arrow, ContourArrow, LineArrow, Triangle, ContourTriangle,
Square, ContourSquare, HalfSquare, Circle, ContourCircle, HalfCircle,
Diamond, ContourDiamond, HalfDiamond, Stroke`

**To close (recommended approach):**
1. **Split the test** into 17 per-end-type tests. This isolates which types
   diverge and by how much.
2. **For each failing type**, compare the Rust polygon vertices (from
   `paint_stroke_end()`) against C++ output pixel-by-pixel.
3. **Fix geometry** per type. Key parameters: `ARROW_BASE_SIZE=10.0` (:9),
   `ARROW_NOTCH=0.3` (:11), width/length factors.
4. The "Contour" variants use `inset_polygon()` which may have different
   inset depth than C++.

**Where to start reading C++:** Search `emPainter.cpp` for stroke end type
handling — likely a switch/case on end type enum near the line painting code.

## 12. Priority ranking for closing gaps
